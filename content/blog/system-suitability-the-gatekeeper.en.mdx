# System suitability: the gatekeeper you can't re-inject past

System suitability testing is the moment a chromatographic run earns the right to
produce reportable results.

It asks a narrow but crucial question:

**Is the complete analytical system performing adequately for this run, before we
trust the samples?**

The "system" is not only the instrument. It includes the column, detector, mobile
phase, gradient, autosampler, standards, integration settings, analyst setup,
equilibration, and the validated method conditions. USP &lt;621&gt; treats system
suitability as a generally applicable chromatography control because separation
methods depend on the whole system behaving as expected.

That is why system suitability is a gate. If the gate fails, the sample results
behind it are not simply inconvenient. They may be scientifically unusable.

## What SST proves, and what it does not

System suitability proves that the method's critical performance requirements are
met in the actual sequence being run.

It does not prove that:

- the analytical method was validated correctly;
- the reference standard is qualified if the standard program is weak;
- the sample preparation was correct;
- the instrument is calibrated for all uses;
- the batch is acceptable;
- a failed run can be repaired by more injections.

Validation, calibration, qualification, sample preparation, and system suitability
answer different questions. They work together, but one cannot replace the others.

A validated method still needs system suitability every routine run because columns
age, mobile phases change, detectors drift, pumps leak, autosamplers misbehave,
standards degrade, and integration can be wrong. A run can fail today even if the
method was excellent during validation.

## The common SST signals

The exact criteria must come from the approved method, monograph, or justified
procedure. Common chromatographic checks include:

| SST check | What it protects | Typical failure meaning |
| --- | --- | --- |
| Resolution of a critical pair | Peak separation and specificity | Co-elution risk, unreliable quantitation |
| %RSD of replicate standard injections | Injection/system precision | Injector, integration, equilibration, or preparation variability |
| Tailing factor or symmetry | Peak shape and integration reliability | Column deterioration, secondary interaction, method condition drift |
| Theoretical plates or efficiency | Column/system performance | Column aging, poor installation, extra-column effects |
| Retention time or relative retention | Method consistency | Flow, composition, temperature, gradient, or column changes |
| Signal-to-noise | Sensitivity for low-level tests | Detector noise, weak response, poor sample/standard preparation |
| Bracketing/check standard recovery | Run stability over time | Drift after the initial SST passed |

Do not add parameters just to look rigorous. The strongest SST criteria are linked
to method-critical performance. If resolution is the reason the assay is
stability-indicating, resolution belongs in routine suitability. If injection
precision drives the reported assay value, replicate standard %RSD matters. If an
impurity is near a reporting threshold, sensitivity and signal-to-noise may matter.

## SST starts before the first sample

For most HPLC/GC sequences, system suitability is evaluated before sample results
are accepted. That usually means:

- prepare the SST or standard solution according to the written method;
- equilibrate the column and system as required;
- inject the required standard or suitability preparation;
- evaluate every required SST criterion;
- review chromatograms and integration, not only the result table;
- proceed to samples only when all gates pass.

The sequence design should make this obvious. An auditor should be able to see
which injections were suitability injections, which were samples, where brackets
were placed, and which results are covered by each passing check.

If the system is not suitable, stop and investigate. Do not keep injecting until a
number looks acceptable and then pretend the earlier injections never happened.

## The re-injection trap

The classic data-integrity failure is "re-inject until it passes."

FDA's data-integrity guidance is especially direct about this area. System
suitability should follow established written procedures, and actual samples
should not be used in test, prep, or equilibration runs as a way to disguise
testing into compliance. FDA's CGMP laboratory-control Q&A also warns that
finished dosage forms or APIs that have not been qualified as reference standards
should not be used for system suitability testing, and that even qualified material
from the same batch as the samples should not be used for that purpose.

The problem is not that every re-injection is forbidden. The problem is
uncontrolled repetition without a documented scientific reason.

Legitimate actions may include:

- stopping after an SST failure;
- documenting the failure and all original data;
- identifying an assignable cause such as air bubbles, leak, incorrect mobile
  phase, failed column equilibration, wrong vial, or instrument fault;
- correcting the cause;
- starting a new sequence according to the procedure;
- explaining which data are invalidated and why.

Weak actions include:

- deleting failed suitability injections;
- labeling real samples as "trial" or "equilibration" injections;
- changing integration only after seeing a failing result;
- repeating injections until %RSD passes without a defined rule;
- using sample injections to decide whether the system is ready;
- accepting sample results from a sequence whose required SST failed.

The audit trail will show the difference. So will the sequence table, injection
history, method version, processing method, and analyst notes.

## Bracketing protects the whole run

A passing SST at injection one does not prove the system remained suitable at
injection eighty.

Long sequences need bracketing standards or periodic check standards when required
by the method or justified by risk. Bracketing confirms that the system stayed in
control across the period where sample results were generated.

When a bracket fails, the question becomes: **which sample results are covered by
the last passing suitability check?**

A practical response may include:

- identify the time and injection range between the last passing and first failing
  bracket;
- review trends in response, retention time, resolution, and peak shape;
- check whether drift was gradual or sudden;
- assess whether samples after the last passing bracket are invalidated;
- investigate likely causes such as column degradation, mobile-phase change,
  evaporation, leak, autosampler temperature, lamp instability, or integration
  changes;
- document the scientific rationale for any accepted or rejected results.

Bracketing is not bureaucracy. It is how you avoid pretending that an early pass
covers a system that later drifted out of control.

## Failure does not automatically mean OOS

A failed SST is not the same thing as a product OOS result. It is usually a run
validity problem.

If suitability fails before samples, sample results are generally not reportable
because the system never earned the right to test them. If suitability fails after
samples, such as a failed bracket, the affected results may be invalidated based on
the predefined procedure and the investigation.

FDA's OOS guidance recognizes that some methods have system suitability
requirements and that systems not meeting them may invalidate the analytical run,
provided original data are retained and the explanation is documented.

The key is discipline:

- retain the failed data;
- document the event immediately;
- determine whether there is an assignable laboratory/system cause;
- define the affected result set;
- decide whether retesting is scientifically justified;
- follow the approved OOS or laboratory investigation procedure where applicable.

Rejecting or invalidating data is never a casual convenience. It needs evidence.

## How to investigate a failed SST

A useful SST investigation is fast, factual, and specific. It does not start with
"analyst error" as the default answer.

For high %RSD, check:

- air bubbles, leaks, injector precision, needle wash, syringe condition;
- vial fill volume, cap/septum condition, evaporation, carryover;
- standard preparation, dilution, mixing, stability, and vial sequence;
- column equilibration and temperature stability;
- integration consistency and peak assignment;
- detector response stability.

For poor resolution, check:

- mobile-phase composition and pH;
- gradient preparation and dwell volume;
- column age, lot, dimensions, installation direction, and storage;
- temperature, flow rate, and equilibration time;
- guard column or frit blockage;
- unexpected co-elution or degradation in the suitability solution.

For tailing or poor efficiency, check:

- column contamination or deterioration;
- sample solvent mismatch;
- injection volume;
- pH or ionic strength;
- tubing, fittings, voids, and extra-column volume;
- integration settings that mask peak shape problems.

For retention-time drift, check:

- pump mixing and flow accuracy;
- mobile-phase evaporation or composition error;
- temperature control;
- gradient timing;
- column equilibration;
- leaks or air in the pump.

The conclusion should state what failed, what was reviewed, what cause was found
or not found, what data were affected, what corrective action was taken, and
whether repeat testing is justified.

## Trend SST instead of waiting for failure

SST data are more useful than many labs realize. They are routine health signals
for the method and instrument.

Trend parameters such as:

- critical-pair resolution;
- replicate standard %RSD;
- retention time;
- tailing factor;
- theoretical plates;
- standard response factor;
- bracket recovery;
- signal-to-noise for low-level methods.

Trends can reveal a column declining before it fails, an injector becoming
variable, a detector losing sensitivity, a mobile-phase preparation problem, or a
method that is too fragile for routine use.

This is where system suitability connects to method lifecycle thinking. ICH Q14
frames analytical procedures as lifecycle assets. If routine SST data show
recurring fragility, the answer may be method improvement, robustness review,
training, clearer procedure controls, or change control, not endless run-by-run
firefighting.

## What reviewers should look for

During second-person review or QA review, do not only verify the final SST result.
Look at the story:

- Were the required SST injections run before applicable samples?
- Were the correct standard or SST preparations used?
- Were criteria pulled from the approved method or monograph?
- Did every required parameter pass?
- Were chromatograms and integration reviewed?
- Were failed or aborted injections retained and explained?
- Were any sample injections labeled as trial, prep, equilibration, or system
  check?
- Were processing methods changed after failures?
- Were bracket failures handled according to procedure?
- Were retests predefined, justified, and documented?
- Are repeated SST failures trended into CAPA or method lifecycle review?

This review catches the real risk: not only whether the run passed, but whether
the pass was earned honestly.

## Common weak spots

System suitability programs often fail in predictable ways:

- SST criteria are copied from a generic template instead of tied to method risk.
- The method does not define what happens when a bracket fails.
- Analysts keep equilibrating and injecting without recording failed attempts.
- Sample injections are used to prove the system is ready.
- Failed injections disappear from the report package.
- The lab reviews only summary tables and misses bad integration.
- %RSD passes, but resolution of the critical pair is marginal.
- Reference standards are expired, unqualified, or from the same batch as samples.
- Trending is not performed, so recurring method fragility looks like isolated
  events.
- Every failure is closed as training, even when the column, method, or instrument
  is the real cause.

The fix is usually not more paperwork. It is clearer sequence rules, better
scientific criteria, complete data review, and honest investigation.

## Key takeaway

System suitability is the run-level proof that the chromatographic system is fit
to generate reportable data today. It is not validation, not calibration, not a
shortcut, and not a number you can chase by re-injecting.

When SST is designed from method-critical performance, executed before samples,
bracketed where needed, reviewed with the audit trail, and trended over time, it
becomes one of the lab's strongest defenses against bad data.

When it is treated as a hurdle to get around, it becomes one of the fastest ways
to create a data-integrity finding.
