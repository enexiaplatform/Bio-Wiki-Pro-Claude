# Database Migrations (Drizzle)

Schema lives in `shared/schema.ts` (+ `shared/models/auth.ts`). Migrations are
versioned SQL files in `migrations/`, generated by Drizzle Kit.

## ⚠️ Production rule

**Do NOT run `npm run db:push` against a production database with data.**
`db:push` diffs and force-applies the schema with no migration history — it can
drop/alter columns and lose data. Use generated migrations in any environment
that holds real data.

- `db:push` → local dev / throwaway DBs only (fast iteration).
- `db:generate` + `db:migrate` → staging & production.

## Commands

```bash
# 1. After changing shared/schema.ts, generate a migration:
npm run db:generate            # writes migrations/NNNN_<name>.sql + meta/

# 2. Apply pending migrations to the DB in DATABASE_URL:
npm run db:migrate

# (dev only) push schema directly without a migration file:
npm run db:push
```

`DATABASE_URL` must be set (generate reads the config but does not connect;
migrate connects and applies).

## Baseline

`migrations/0000_baseline.sql` captures the current schema (7 tables: `users`,
`sessions`, `purchases`, `leads`, `quote_requests`, `processed_stripe_events`,
`content_entries`). On a clean database, `npm run db:migrate` creates all of
them.

## Workflow

1. Edit `shared/schema.ts`.
2. `npm run db:generate` (review the generated SQL — commit it).
3. Commit the migration file with the code change.
4. Deploy → run `npm run db:migrate` against the target DB (CI step or release task).

## Notes

- Migration files are committed to git — they are the source of truth for DB shape.
- Never hand-edit an already-applied migration; create a new one.
- The `migrations/meta/` journal tracks which migrations have run — keep it committed.
